# 后台可视化页面生成 Skill 步骤记录

## 目标

根据需求描述快速生成后台管理系统的可视化页面，并让每条明确规则都有可浏览器打开的页面用于验收。

## 提炼说明

本文件保留用户逐步提出的原始规则和预览验收记录。最终可复用的后台可视化生成 skill，需要从这些记录中提炼出：

- 框架规则：后台页面默认结构和通用组件约束。
- 视觉规则：颜色、尺寸、风格等统一规范。
- 流程规则：当用户提出后台需求时，Codex 应如何确认菜单归属、如何生成页面、如何避免擅自补内容。

提炼版草案见：

- `work/admin-visual-generation-skill-draft.md`

## 已确认规则

### 规则 00A：后台所有需要颜色填充的色值严格按照灰色色阶

生成后台可视化页面时，所有需要颜色填充的 UI 色值必须严格使用灰色色阶：

- 页面背景、顶部栏、侧边栏、Logo 占位底色、菜单底色、时间底色、边框、悬停底色等填充色只能使用灰阶。
- 不使用绿色、蓝色、紫色、橙色等彩色填充。
- 文字颜色也优先使用灰阶，确保整体后台界面克制、统一。
- 按钮灰阶不要过深也不要过浅，功能按钮默认使用中浅灰底色，避免深灰实底抢占页面视觉重心，也避免浅到和普通按钮区分不足。
- 功能按钮不使用额外深色边框强调，边框保持常规灰线；通过底色和字重区分主次。
- 按钮默认态与 hover 态不能撞色：普通按钮默认白底，hover 使用 `gray-50`；功能按钮默认使用 `gray-200`，hover 使用 `gray-300`。
- 如果后续用户指定品牌色或业务状态色，需要先确认是否允许突破灰阶规则。

### 规则 000：只渲染用户已明确描述的步骤

生成后台可视化页面时，只能渲染用户已经明确说到的页面组成和功能。不要主动补充左侧菜单、中间卡牌、右上角操作区、按钮、搜索框、业务功能、示例数据等未被用户描述的内容。

- 可以提出发散建议，但必须先询问用户是否采纳。
- 发散建议在用户明确采纳前，不进入规则记录和页面渲染。
- 如果某个页面区域尚未被描述，保持空白或仅保留必要页面画布，不放置推测性内容。

### 规则 000A：未指定具体内容时按语义情景推理填入

当用户已经明确需要某个页面、模块、字段或组件，但没有给出具体文案、示例值、占位数据或默认值时：

- 不使用「配置项 A / B / C」「示例名称」「测试数据」这类机械占位。
- 根据页面类型、字段含义、业务语义和上下文场景，推理并填入合理内容。
- 表格数据、表单默认值、下拉选项、说明文案、按钮文案等都适用该规则。
- 推理填入只能补齐已确认页面所需内容，不能借机新增未确认模块或功能。
- 规则记录只沉淀“按语义情景推理填入”的方法，不记录具体业务值、专有名称或活动文案。

### 规则 001：左侧菜单栏品牌区加平台 Logo 和平台名称

生成后台页面时，左上角必须包含平台品牌识别区，并且品牌区归属于左侧菜单栏视觉区域：

- 放置平台 Logo。
- Logo 右侧展示平台名称。
- 品牌识别区应位于页面首屏左上角。
- 品牌区与左侧菜单列使用同一灰阶底色，默认使用 `gray-600`。
- 品牌区高度与顶栏高度一致，桌面端默认为 `64px`。
- 平台名称在深灰底上使用浅色文字，保证可读性。
- Logo 与平台名称需保持清晰、稳重、后台系统风格，避免装饰化过重。
- 无真实 Logo 时，默认使用首字母、抽象几何标识或产品简称作为占位 Logo。
- 平台名称下方不要增加副标题、小字或后台说明，除非用户明确要求。

### 规则 002：左侧加两个菜单，每个菜单里面加两个子菜单

生成后台页面时，左侧必须包含菜单区域：

- 左侧放置 2 个一级菜单。
- 每个一级菜单下放置 2 个子菜单。
- 一级菜单支持展开和收起交互。
- 左侧一级菜单默认全部收起。
- 左侧一级菜单采用手风琴联动：当已有菜单展开时，再展开新的一级菜单，需要自动收起已有展开菜单。
- 点击已展开的一级菜单时，收起当前菜单。
- 左侧菜单展开 / 收起需要过渡动画，不能瞬间切换。
- 一级菜单需要添加底色，用于和子菜单形成层级区分。
- 子菜单需要具备悬浮状态和选中状态。
- 子菜单悬浮状态使用灰阶浅底色和更深文字色。
- 子菜单选中状态使用更深一级灰阶底色、深色文字和适度加粗。
- 点击子菜单时只切换选中态，不主动生成对应内容页。
- 左侧菜单默认采用紧凑密度，一级菜单组间距和子菜单间距不要过大。
- 左侧菜单整列需要加灰阶底色，默认可使用 `gray-600`；平台品牌区也属于左侧菜单栏并使用同一底色。当菜单列底色加深时，子菜单文字、悬浮态和选中态必须同步调整对比度，避免看不清。
- 只展示菜单结构，不主动添加菜单图标、徽标、数量；子菜单选中态不联动生成对应内容页或其他功能。
- 菜单名称在用户未指定时，可使用通用占位文案，例如「一级菜单一」「子菜单 1-1」。

### 规则 002A：顶栏加入功能模块搜索

用户明确要求时，在顶栏品牌区右侧、右侧信息区左侧加入功能模块搜索：

- 搜索框提示词使用「搜索后台功能」。
- 搜索框左侧需要有搜索 icon。
- 搜索框用于模糊搜索左侧已有功能模块。
- 搜索框不生成独立搜索结果页。
- 输入关键字时，在搜索框下方展示所有匹配预览结果。
- 模糊搜索需要支持非完整输入，例如输入「子菜单」可以展示所有子菜单结果。
- 预览结果展示功能名称和路径，例如「一级菜单一 / 子菜单 1-1」。
- 点击预览结果后，自动展开对应一级菜单、选中对应子菜单，并收起搜索结果。
- 无匹配结果时，在搜索框下方显示「无匹配模块」空状态。
- 搜索框位置应位于平台名称右侧附近，不挤压右上角时间和用户信息。

### 规则 002B：内容区加面包屑

用户要求面包屑时，在主内容区顶部、筛选区或页面主体内容上方展示当前功能路径：

- 面包屑展示当前一级菜单 / 子菜单路径，例如「运营管理 / 补水活动」。
- 父级路径使用弱化文字，当前页面使用更深文字和适度加粗。
- 点击左侧子菜单或搜索结果定位菜单时，面包屑需要同步更新。
- 面包屑只表达当前位置，不主动新增返回按钮、标签页、页面标题或额外操作。

### 规则 002C：可关闭多页签导航

用户要求页签或多标签页导航时，在主内容区顶部展示已打开功能页签，通常放在面包屑上方：

- 该组件命名为「可关闭多页签导航」或「Tab 页签导航」。
- 默认只打开当前功能页签，不主动生成未访问功能页签。
- 用户点击左侧子菜单或搜索结果进入功能时，如果该功能页签不存在，则新增页签并激活；如果已存在，则只激活已有页签，不重复新增。
- 当前页签需要高亮显示。
- 页签显示子菜单 / 功能名称，关闭入口使用 `×`。
- 点击页签时，同步切换左侧菜单选中态、一级菜单展开态和面包屑。
- 关闭当前激活页签时，需要自动切换到相邻页签，优先右侧，没有右侧则切左侧。
- 至少保留一个已打开页签，除非用户明确允许关闭到空状态。
- 页签只表达已打开功能，不主动生成页面内容。

### 规则 003：右上角加当前后台对应国家的实时时间展示

生成后台页面时，顶部导航栏右上角必须展示当前后台对应国家的实时时间：

- 时间展示位置为页面右上角。
- 时间内容根据当前后台对应国家或地区的时区计算。
- 展示格式包含 UTC 偏移量、日期和时间，例如「UTC+05:30 : 2026-07-01 14:11:12」。
- 时间需要实时刷新，至少每秒更新一次。
- 时间展示尺寸要克制，不要过大抢占顶栏视觉重心。
- 时间展示需要添加圆角矩形底色。
- 未指定具体国家或时区时，可使用截图示例的「UTC+05:30」作为占位时区。
- 只生成时间展示，不主动添加语言选择、账号名称、用户菜单、修改密码、退出登录等截图中的其他元素。

### 规则 004：右上角加用户 Logo 和用户昵称

生成后台页面时，顶部导航栏右上角必须展示当前用户信息：

- 展示 `👤` 用户 Logo。
- 展示用户昵称。
- 用户 Logo 和用户昵称放在右上角区域，与实时时间并列展示。
- 右上角用户信息默认不使用头像图片，也不使用圆形头像占位。
- 未提供真实昵称时，使用通用占位昵称，例如「用户昵称」。
- 用户昵称字号要小于或弱于主要品牌名称，避免抢占顶栏视觉重心。
- 用户明确要求时，用户 Logo 和昵称区域支持下拉菜单。
- 用户下拉菜单包含「修改密码」「退出登录」两个操作项。
- 用户明确要求语言选择时，语言入口放在用户 Logo / 昵称下拉框内部，不在顶栏单独生成语言控件。
- 用户下拉内语言选择默认显示当前语言，例如「中文」；点击后展开语言选项，例如「中文」「English」；选择后更新当前语言显示并收起语言选项。
- 用户下拉交互流程：点击用户 Logo / 昵称展开，再次点击收起，点击外部收起，按 Esc 收起。
- 用户明确要求补齐后续内容时，「修改密码」打开修改密码弹窗，包含旧密码、新密码、确认新密码、取消、确认修改。
- 用户明确要求补齐后续内容时，「退出登录」打开退出确认弹窗，包含确认文案、取消、确认退出。
- 弹窗关闭流程包含：取消、关闭按钮、点击遮罩、按 Esc。
- 不主动添加下拉箭头、额外菜单项、二次确认后的真实接口逻辑，除非用户明确要求。

### 规则 005：后续所有生成都要兼顾美感

生成后台可视化页面时，不能只机械满足功能点，还要兼顾整体美感：

- 控制字号、间距、圆角、边框、留白和组件比例。
- 顶栏、菜单、用户信息、时间等组件要对齐清楚，视觉重心稳定。
- 不因为动态时间、长昵称、菜单展开收起等状态造成挤压、重叠或跳动。
- 在不新增用户未确认内容的前提下，允许为美感调整布局比例和视觉细节。

### 规则 005A：UI 长宽和间距要写成明确数值

生成后台可视化页面时，涉及长宽、间距、圆角、字号、边框等视觉细节，要沉淀成明确数值规则，不能只写“紧凑”“好看”“适中”。

- 顶栏高度：桌面端 `64px`，移动端 `56px`。
- 左侧菜单宽度：桌面端 `232px`。
- 顶栏左右内边距：桌面端 `20px`，移动端 `14px`。
- 顶栏左右区域间距：桌面端 `16px`，移动端 `10px`。
- 顶栏右侧内部组件间距：桌面端 `12px`，移动端 `8px`。
- 侧边栏内边距：`16px 12px`。
- 顶栏三段布局：左侧品牌区 `232px`，功能模块搜索 `minmax(220px, 260px)`，右侧信息区 `minmax(0, 1fr)`。
- 移动端顶栏三段布局：品牌区 `auto`，功能模块搜索 `minmax(140px, 180px)`，右侧信息区 `minmax(0, 1fr)`。
- 功能模块搜索框高度：桌面端 `34px`，移动端 `30px`。
- 功能模块搜索框宽度：桌面端 `220px-260px`，移动端 `140px-180px`。
- 功能模块搜索框圆角：`6px`。
- 功能模块搜索框内边距：桌面端 `0 10px 0 30px`，移动端 `0 8px 0 28px`。
- 功能模块搜索框字号：桌面端 `13px`，移动端 `12px`。
- 搜索 icon 左偏移：桌面端 `10px`，移动端 `9px`。
- 搜索结果浮层与搜索框垂直距离：桌面端 `8px`，移动端 `6px`。
- 搜索结果浮层最大高度：桌面端 `220px`，移动端 `180px`。
- 搜索结果项最小高度：`44px`。
- 平台 Logo 尺寸：桌面端 `36px × 36px`，移动端 `32px × 32px`。
- 平台 Logo 圆角：桌面端 `8px`，移动端 `7px`。
- Logo 与平台名称间距：桌面端 `12px`，移动端 `10px`。
- 平台名称字号：桌面端 `16px`，移动端 `15px`。
- 右上角时间最小高度：桌面端 `32px`，移动端 `28px`。
- 右上角时间最大宽度：桌面端 `320px`，移动端 `180px`。
- 右上角时间内边距：桌面端 `0 12px`，移动端 `0 8px`。
- 用户 Logo 尺寸：桌面端 `16px × 16px`，移动端 `15px × 15px`。
- 用户 Logo 字号：桌面端 `12px`，移动端 `11px`。
- 用户昵称最大宽度：桌面端 `140px`，移动端 `72px`。
- 用户昵称字号：桌面端 `13px`，移动端 `11px`。
- 用户下拉宽度：桌面端 `136px`，移动端 `128px`。
- 下拉项高度：桌面端 `34px`，移动端 `32px`。
- 一级菜单高度：最小 `36px`。
- 一级菜单组间距：`8px`。
- 子菜单顶部间距：`8px`，避免第一个子菜单和一级菜单贴得太近。
- 子菜单左侧缩进：`12px`。
- 子菜单项高度：最小 `30px`。
- 子菜单悬浮动画：`background 160ms ease`、`color 160ms ease`。
- 子菜单展开最大高度：两项子菜单默认 `80px`。
- 子菜单展开 / 收起动画：`max-height 220ms ease`、`opacity 180ms ease`、`transform 220ms ease`。
- 菜单箭头旋转动画：`transform 220ms ease`。
- 弹窗宽度：`min(420px, 100%)`。
- 弹窗内容内边距：桌面端 `18px`，移动端 `14px`。
- 输入框高度：`36px`。
- 筛选 / 输入控件统一基础尺寸体系：基础尺寸 1 为 `256px`，字段间距 `16px`。
- 尺寸 1：`256px`，作为默认基础尺寸，适用于状态、类型、名称、ID、手机号等常规筛选和输入。
- 尺寸 2：`528px`，适用于较长 placeholder、组合编号、邮箱等。
- 尺寸 3：`800px`，适用于长业务对象、长路径、较长说明输入。
- 尺寸 4：`1072px`，适用于超长提示词或复合条件输入。
- 尺寸 5：`1344px`，适用于跨行主搜索或大段长文本筛选。
- 尺寸 6：`1616px`，适用于需要完整展示超长提示词的宽屏场景。
- 同一筛选区内控件必须按尺寸 1-6 选择，不允许因文案长度临时写任意宽度；提示词过长时优先升级到更大尺寸，不压缩字号。
- 按钮最小宽度：`76px`，按钮高度：`34px`。
- 常规边框宽度：`1px`。
- 常规控件圆角优先使用 `6px` 或 `8px`。

### 规则 006：后续交互和流程需要补齐

生成后台可视化页面时，如果用户明确要求某个组件具备交互，不能只画静态样式，需要补齐基础交互流程：

- 明确触发方式，例如点击、悬停、输入、切换。
- 明确展开、收起、取消、关闭等状态变化。
- 明确常见关闭方式，例如再次点击、点击外部、按 Esc。
- 展开 / 收起类交互默认使用 `180ms-220ms` 的过渡动画，避免切换过快。
- 只补齐与用户明确要求相关的交互，不延展出额外业务流程。

### 规则 007：复刻具体业务页面时只记录抽象实现逻辑

当用户提供具体业务页面并要求复刻时：

- 预览页可以复刻原页面的布局结构、组件密度、交互流程和信息组织方式。
- skill 记录不能写入具体业务名称、专有字段、真实业务规则或原始业务文案。
- 需要抽象成通用页面类型，例如配置类管理页、列表查询页、详情编辑页等。
- 配置类管理页优先沉淀为「筛选区 + 工具栏 + 数据表格 + 右侧新增 / 编辑抽屉」结构。
- 用户要求状态筛选时，在筛选区增加「状态」下拉字段，默认值为全部，选项与表格状态值保持一致。
- 状态筛选点击查询后联动表格行显示；点击重置后恢复全部行，并清空当前批量选择状态。
- 用户要求给表格新增字段列时，按用户指定位置插入；如果用户指定“在某列前 / 后”，必须严格保持列顺序。
- 根据需求生成表格示例数据时，优先识别字段中的枚举数量，选择枚举最多的字段作为行数基准。
- 如果某字段有 `N` 个枚举且 `N >= 8`，默认生成 `N` 行数据，并保证每个枚举值至少出现一次。
- 如果枚举数量低于 `8`，默认仍生成 `8` 行数据；枚举值可随机或轮询填入，但必须保证所有枚举值都有展示。
- 多个字段都有枚举时，优先保证枚举最多字段完整覆盖，其他枚举字段在 8 行或更多行中尽量均匀分布。
- 表格示例名称优先使用用户提供材料中的真实命名；真实命名不足时，根据页面业务语义推理合理名称，避免使用「配置项 A / B / C」这类机械占位。
- 规则记录时只沉淀命名策略，不记录具体业务名称或专有活动名称。
- 用户要求表格补充状态与操作时，在表格末尾增加「状态」「操作」列；状态用紧凑标签展示，操作用文字按钮承载编辑、启用、停用等行级动作。
- 状态与操作需要补齐基础交互：编辑打开对应抽屉，启用 / 停用切换当前行状态和操作文案。
- 用户要求批量操作时，在表格序号列前增加批量选择列；表头提供全选复选框，行内提供单选复选框。
- 批量按钮放在工具栏主按钮右侧，按用户要求提供对应批量动作，例如批量启用、批量停用。
- 全选需要支持半选状态；批量动作只影响已勾选行，并同步行状态标签和行级操作文案。
- 在灰色色阶规则生效时，状态标签和操作按钮仍使用灰阶，不直接使用红绿蓝等业务色。
- 右侧抽屉默认关闭，不在首屏自动展开；只有用户点击新增、编辑等明确触发操作后才打开。
- 抽屉内复杂配置默认按基本信息、参与条件、规则配置、前台展示拆分；具体字段优先映射到这四类，不随意新增一级类目。
- 参与资格、开关和条件矩阵归入参与条件；触发策略、阈值、次数、触发范围、区间矩阵、礼金、奖励、权益等规则相关属性归入规则配置；客户端展示名称、多语言、宣传文案、展示素材归入前台展示。
- 前台展示默认沿用「语言页签 + 展示字段区 + 自定义说明」结构：语言页签在顶部，展示字段区左侧放文案 / 排序字段，右侧放活动中心 banner 图和详情图，自定义说明在下方用富文本编辑区承载。
- 当前活动前台展示字段为活动名称（前台）、banner 图排序、活动中心 banner 图、详情图、自定义说明；不再生成独立的活动宣传文本字段，原活动宣传文本位置由 banner 图排序替代。
- 活动名称（前台）属于客户端露出的展示名称，不等同于后台列表活动名称；后台活动名称用于管理识别。
- 前台展示字段区需要控制整体高度与垂直重心：左侧多行输入与右侧图片上传项应垂直居中对齐，图片槽不贴顶，内容区可设置稳定最小高度避免视觉松散或压迫。
- 前台展示字段区左侧输入列不强制撑满整行，应使用克制最大宽度；右侧活动中心 banner 图和详情图上传位需要等比放大到易识别尺寸，并保持两张图同宽同高。
- 弹窗展示属于前台展示配置；当需求出现活动弹窗、弹窗排序、弹窗按钮跳转或弹窗图时，应放在前台展示模块内。基础字段包括弹窗是否展示开关、弹窗展示顺序、弹窗按钮跳转类型（跳转到指定页面 / 外链）、跳转目标和弹窗图片上传；展示顺序说明为数字越大越靠前。
- 自定义说明卡片中如果存在段落标题输入框，输入框上方必须显式展示字段名「自定义说明段落标题」，不能只放一个无标签输入框。
- 用户标签、风险排除、会员资格等用于判断用户是否可参与的对象条件，应放入参与条件矩阵；不要在规则配置中重复做“限定对象”类基础字段。
- 同一规则块内部若同时包含整体规则和明细参数，应拆成轻量小模块标题，例如基础规则配置、区间参数配置，不额外套大卡片。
- 礼金配置应放在规则配置下作为子模块；礼金倍率、礼金适用范围及其联动范围值放入礼金配置，不混入基本信息或基础规则配置。
- 当业务当前只有一种类型或无需用户选择类型时，不展示类型下拉字段，也不在列表中展示类型列。
- 当页面存在全局适用前提、参与资格或规则范围时，应在基础信息上方用克制提示条展示，避免把前提散落到多个字段里。
- 当配置逻辑涉及适用对象、触发条件、执行结果和限制条件时，可在顶部提示条中用一段简洁说明串联完整逻辑，帮助用户先理解配置如何生效；skill 记录只保留“逻辑说明结构”，不记录具体业务文案。
- 顶部逻辑说明必须区分限制作用对象：触发限制用于判断规则是否生效，发放后要求用于说明奖励、礼金、权益等到账后的使用或提现条件，不能把两者混写成同一个对象的限制。
- 字段存在防呆上限、校验边界或保存拦截规则时，优先放在字段名后的问号悬浮提示中说明；例如金额上限、余额口径、命中口径等，不额外占用表单说明行。
- 余额口径类提示应完整列明参与计算的账户或金额项，例如锁定钱包、可提现钱包、在途金额等，避免只写泛化的“余额”。
- 区间类配置需要说明边界规则和命中规则：区间不可重叠，可断开；断开区间不命中规则。区间公式可放在字段名后，详细约束放在问号悬浮提示里。
- 发放后使用范围与触发规则范围是不同语义：前者描述奖励、礼金、权益到账后的可使用范围，优先放在基本信息或奖励基础属性中；后者描述规则是否命中的范围，放在规则配置中。
- 范围类字段应成对展示范围类型和范围值；范围类型决定范围值的选项集合。
- 范围类型存在“无限制”选项时，默认隐藏第二级范围值；选择指定对象后再展示第二级范围值，并按一级类型重置默认选项。
- 当范围值支持选择多个对象时，第二级范围值控件应使用多选组件，展示已选摘要，并在一级范围变化时重置为该范围下的合理默认选项。
- 多选下拉不要直接暴露原生复选框样式；应使用整行可点击选项、定制灰阶勾选标记和简洁已选摘要，避免视觉粗糙拥挤。
- 条件配置行需要采用紧凑表格化布局：表头展示「条件类型 / 条件值 / 操作」，下一行展示条件类型下拉、运算符下拉、数值输入和单位；右侧操作列固定较窄宽度，用于新增或删除条件。
- 条件配置组件的表头高度、内容行高度、列间距和输入控件高度必须统一，不使用普通数据表格样式套用，避免显得粗糙拥挤。
- 条件类型需要驱动条件值控件联动：数值类条件使用比较运算符和带单位输入；标签、渠道、等级等枚举类条件使用等于 / 不等于和下拉选择；ID 列表类条件使用可换行输入并提示换行分隔。
- 事件后 N 天、注册后 N 天、首充后 N 天等时间偏移类条件，使用比较运算符和带时间单位的数值输入表达。
- 范围类运算符（如“介于”）需要将条件值切换为双输入区间，中间使用 `~` 分隔，两端都保留单位展示；切回单值运算符时恢复单输入。
- 区间阈值触发类配置应使用规则矩阵表达：左侧是对象区间或分层区间，中间是触发阈值，右侧是发放值、附加要求和触发次数；每一行代表一条完整规则。
- 区间字段需要在表头或字段说明中明确边界包含关系，例如最小值是否包含、最大值是否排除，避免配置人员误解。
- 表格列或字段存在复杂口径、命中边界、计算范围、排除条件等说明时，优先在字段名或表头后放带底色的问号 tooltip；悬浮展示完整说明，避免把长解释直接铺在表格里。
- Tooltip、下拉层、浮层说明必须高于当前组件层级展示，并检查父容器 `overflow` 是否会裁切；必要时为承载区开启可见溢出或使用独立浮层层级。
- 当规则存在可触发次数、可领取次数、可补次数等次数限制时，次数列必须作为规则矩阵的独立列展示，不能隐藏在说明或折叠区域中。
- 当同时存在整体累计次数和单行规则次数时，整体累计次数放在规则矩阵上方作为独立字段，单行次数保留在矩阵列中，二者不要混用。
- 作用于触发判断的整体限制、范围等字段应放在规则块的基础规则配置中，不放在每一行明细规则里；如果倍率、比例等字段描述的是发放后要求或活动基础属性，则可放在基本信息中。
- 规则块的基础规则配置字段较多时，使用两列网格布局，字段顺序按业务触发顺序、限制优先级和配置阅读路径排列。
- 规则块基础配置的两列字段应使用标准字段结构：标签在上、控件在下，单位内置或贴近控件右侧；不要把标签、输入框、单位强行挤在同一行导致左右列起点不齐。
- 内容编辑器、富文本工具条、自定义说明、短操作按钮等局部编辑场景不强制撑满父容器；应使用内容自适应宽度或合理最大宽度，避免控件被拉得过长。
- 如果范围字段限定的是规则矩阵整体生效范围，应放入规则块的基础规则配置中，不放在页面基础信息里。
- 字段解释、特殊值含义、配置影响等短说明在能放下时优先放在字段标签后方，不单独占用控件下方空间；只有长说明、复杂说明或会造成拥挤时才另起一行。
- 如果某个配置项对整组规则统一生效，应放在基础信息或规则矩阵上方，不要在每一行规则里重复展示。
- 当规则目标依赖用户阶段或资格状态时，将该资格状态放在参与条件中表达，不混入策略区间表。
- 具有条件切换、类型切换、范围切换、单位切换的页面，需要记录为字段联动逻辑，不记录具体业务计算规则。
- 当字段语义是“是否开启 / 是否启用 / 是否允许”等布尔控制时，优先使用开关组件表达，不使用两个单选项表达；开关文案和受控配置区块需要联动显示或隐藏。
- 活动、任务、权益、配置等同类对象较多时，左侧菜单应按稳定业务域和管理模块组织，不按每一种具体对象单独拆菜单；对象差异用列表字段、类型筛选、模板或新增 / 编辑表单承载。
- 活动类后台默认信息架构为「运营管理 / 活动管理 / 活动列表、活动新增、活动编辑」；左侧菜单选中到「活动管理」，列表页展示所有活动，新增和编辑可用独立页面或抽屉承接。
- 当从单一活动页面升级为统一活动管理时，列表需要保留「活动类型」字段或筛选，用来区分不同活动来源；新增 / 编辑内再根据活动类型加载对应配置项。
- 将原后台已有活动迁入统一活动管理时，左侧不新增该活动的独立菜单；活动列表增加对应类型数据，编辑入口根据活动类型切换到对应配置模板。
- 统一活动列表优先展示跨活动通用字段，当前列顺序为序号、包ID、活动ID、活动名称、活动类型、banner图排序、活动中心banner图、详情图、修改人、时间、状态、操作；序号列保持紧凑宽度，包ID紧跟序号展示。
- 活动列表中的图片字段使用缩略图展示，不只展示“已上传 / 未上传”文本；排序字段需要使用准确字段名，例如 banner 图排序，不用图片字段名替代。
- 只有某个活动类型独有且不适合跨类型展示的字段，不强行放入统一列表列中。
- 活动类型差异化配置应在编辑页或抽屉内按「基本信息、参与条件、规则配置、前台展示」归类；基础字段、条件矩阵、奖励策略、展示素材分别进入对应区块。
- 表单中出现可选择的配置资源、模板资源、素材资源、规则资源等下拉字段时，后台结构中应能找到对应资源的管理入口或来源说明，不能只有选择项而没有维护位置。
- 「客户端页面模板 / 前台页面模板 / 展示模板」属于前台展示资源，负责控制客户端展示版式、素材字段和文案结构；它不参与活动规则命中、金额计算或发放逻辑。
- 当活动编辑中提供客户端页面模板选择时，左侧菜单或相关管理模块中应存在「客户端页面模板管理」入口，用于维护模板列表、模板字段结构和启停状态。
- 模板管理入口或来源确认后，活动编辑的前台展示区才能展示「客户端页面模板」选择项；选择项来源于模板管理中的已启用模板，并可在字段标签后用短说明标注来源。
- 复刻输出仍需严格遵守灰色色阶、明确尺寸、补齐基础交互、只生成已确认内容等已确认规则。

## 发散建议处理记录

### 建议 A：Logo 形态

已采纳：固定使用「图标 + 文字」组合，并允许在无真实 Logo 时使用首字母、抽象几何标识或产品简称作为占位。

不采纳：平台名称下方不增加「Admin Console / 管理后台」等副标题。
